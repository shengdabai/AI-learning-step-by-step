Module 1

先把 GitHub 设置对

初学者最容易忽略的其实不是命令，而是账户安全、认证方式、通知规则和仓库卫生。先把这些做对，你后面学 Git 会顺很多。

个人资料与 Profile README第一印象

补齐头像、Bio、主页、README，让别人一眼看懂你是谁、做什么、项目在哪。

2FA / Passkey / 恢复码必须优先

GitHub 官方长期强调 2FA；恢复码也要保存好，不然丢失认证方式时可能永久失去账户。

SSH Key 或 PAT连接电脑

你在命令行 push / pull 时，需要 SSH key 或 PAT 作为安全身份验证方式。

通知规则减少噪音

把邮件和站内通知调好，不然你会被 review、mention、watch 消息淹没。

本地 Git 配置你的电脑

安装 Git，配置用户名和邮箱，确认默认编辑器与认证方式，这一步属于 Git 而不是 GitHub 页面。

仓库卫生习惯长期收益

README、LICENSE、.gitignore、保护分支、Issue / PR 模板，会决定你以后协作是否清晰。

Module 2

先分清 Git 和 GitHub

很多人一听到 PR、commit、branch 就晕，本质上是边界没分清。只要先把“本地版本控制”和“远程协作平台”拆开，后面就顺了。

概念关系图
你本地电脑 → Git → GitHub → 团队协作
  • Git 负责记录历史、分支、diff、回退。
  • GitHub 负责托管远程仓库、PR、Review、Issue、Actions。
  • AI 帮你改代码，但 Git 和 GitHub 帮你证明、审查和协作。

看图方式：代码先在本地电脑被修改，Git 负责管理这些变化，GitHub 再把这些变化变成可协作的流程。

记忆法：Git 像你手上的版本记录器；GitHub 像整个团队共享的代码基地和审查台。AI 写出来的改动，Git 负责留痕，GitHub 负责协作。
Module 3

编程基础地基：你最该先补哪几块

你不用立刻学一整门计算机课，但至少要知道代码最基本的积木。否则你看 AI 改出来的代码，会像在看一堆无法判断对错的符号。

变量 Variable

变量是“给数据起名字”。名字有了，你后面才能追踪这份数据在哪里被创建、修改和使用。

条件 Condition

条件是“如果……就……”。程序不是永远直线执行，而是根据条件决定走哪条路。

循环 Loop

循环是“对一组东西重复处理”。AI 改列表、表格、批量数据时，经常会涉及循环。

函数 Function

函数是“可重复调用的处理块”。参数是输入，返回值是输出。

最重要的理解模型
输入 Input → 处理 Logic → 输出 Output

绝大多数代码都可以先粗略理解成这三步。变量负责存输入，条件/循环/函数负责处理，返回值、页面显示、文件输出或 API 响应属于输出。

function greet(name) {
  const message = 'Hello, ' + name;
  return message;
}
你最该会看懂的 7 个基础词
数据类型 data type

至少知道字符串、数字、布尔值、数组、对象分别长什么样，很多报错都和数据类型不匹配有关。

参数 parameter

函数吃进去的输入。比如名字、路径、端口、用户输入都可能是参数。

返回值 return value

函数处理完后吐出来的结果。没有返回值时，后面的代码就可能拿不到它想要的东西。

对象 object

一组“键:值”数据。API 返回 JSON 时，你几乎一定会看到对象。

数组 array

一组按顺序排列的值。列表页面、批量任务、循环处理经常用到数组。

作用域 scope

变量在哪个范围里能被访问。很多“变量未定义”本质就是作用域问题。

报错 error

报错不是坏事，它是在告诉你：哪一层出了问题、缺了什么、类型错了还是路径错了。

Module 4

终端与文件系统：Claude Code / Codex 用户必须补的基础

你现在很多工具都跑在终端里，所以“路径、文件、命令在哪执行”必须看懂。不会这层，就会经常不知道 AI 为什么改错目录、为什么找不到文件、为什么命令跑不起来。

文件树和路径到底是什么 
/Users/adam
├── Desktop
│   └── GitHub学习作战地图.html
├── Projects
│   └── my-app
│       ├── package.json
│       ├── src
│       │   └── app.js
│       └── README.md
└── Downloads
  • 绝对路径：从根开始写完整位置，例如 /Users/adam/Desktop。
  • 相对路径：相对于当前目录，例如 src/app.js。
  • 当前工作目录 cwd：你命令现在是站在哪个目录里执行的。
最常见的终端命令
pwd

看你当前站在哪个目录。很多路径错误，第一步就该先看 pwd。

ls

列出当前目录里有什么文件和文件夹。

cd path

切换目录。让 AI 跑命令前，你也要知道它应该站在哪。

mkdir / touch

创建文件夹和文件。

rm

删除文件。危险命令，理解后再用，别盲删。

终端和 Finder 的关系

Finder 是图形界面看文件；终端是文本方式操作文件。它们操作的是同一套文件系统。

你现在最该建立的判断：看到“file not found”“No such file or directory”“Cannot open”时，优先检查当前目录、路径写法、文件是否真的存在，而不是先怀疑 AI 模型本身。
Module 5

Web 基础：前端、后端、浏览器、服务器、API 到底怎么连起来

只要你在做网页、产品页、表单、后台、接口、登录、列表、数据库，就绕不开这层。很多人知道单词，但不知道它们在一条链上各自做什么。

简化理解：浏览器里运行前端，前端向后端/API 发请求，后端从数据库读写数据后再把结果返回回来。

这条链上每个词是什么意思
前端 Frontend

用户直接看到和点击的页面层，比如按钮、表单、列表、弹窗、导航。

后端 Backend

处理业务逻辑、权限、数据读写、校验、接口返回的那一层。

API

前端与后端通信的接口。前端问，后端答，常见数据格式就是 JSON。

HTTP

浏览器和服务器传递请求/响应时最基础的协议。看到 200、404、500 时，本质都和 HTTP 响应有关。

JSON

一种文本数据格式，API 最常用的返回方式之一。你会经常看到对象和数组长成 JSON 的样子。

表单和验证

表单收集用户输入，前端会先做基本验证，后端还需要再验证一次。

一个页面加载发生了什么

浏览器请求 HTML → 下载 CSS/JS → 渲染页面 → 执行交互逻辑 → 可能继续请求 API。

为什么你会看到接口报错

常见原因：路径错、参数错、权限没带、后端异常、网络失败、返回的数据结构和前端预期不一致。

为什么 JSON 值得先学

它是前端、后端、API、配置文件、自动化工具之间最常见的通用语言之一。

Module 6

运行时、依赖、异步、环境变量：AI 编程里最常卡住的工程层

很多人以为代码写出来就结束了，实际上大量问题根本不在“语法”，而在运行环境。你要逐步分清 JavaScript、Node.js、浏览器、依赖包、环境变量、模块系统分别负责什么。

异步 async / await

异步不是“更高级的同步”，而是程序在等待网络、文件、数据库时不阻塞整个流程。你看到 fetch、await、Promise 时，本质上都在处理“结果不会立刻回来”。

async function loadUser() {
  const response = await fetch('/api/user');
  const data = await response.json();
  return data;
}
依赖与 package.json

package.json 是项目的说明书之一，里面通常定义依赖、脚本、项目入口。看到 npm install、npm run dev、npm test 时，通常都和它有关。

  • dependencies：运行项目需要的包。
  • devDependencies：开发时需要的包。
  • scripts：一组可执行命令。
环境变量 env

环境变量是项目运行时读取的外部配置，例如 API Key、数据库地址、端口号。它们通常不应该直接写死在代码里，更不应该提交到公开仓库。

  • 常见名字：API_KEY、DATABASE_URL、PORT。
  • Node.js 中常通过 process.env 读取。
  • 缺少环境变量时，项目常表现为启动失败、鉴权失败、请求失败。
模块与导入导出
一个文件 → export → 另一个文件 import

很多“找不到函数”“模块不存在”“default/export 错误”都和这里有关。你至少要知道：一个文件可以输出东西，另一个文件再把它引进来用。

这一层最常见的报错信号
Cannot find module

依赖没装、路径写错、模块名写错，或当前运行目录不对。

process.env 为空或 undefined

环境变量没配置、变量名不一致，或没有正确加载 .env。

await 相关错误

可能是在不支持的位置用了 await，或者忘了把函数声明成 async。

npm script 找不到

package.json 里没有对应脚本，或者当前目录不是项目根目录。

Module 7

调试与验证：不要把“模型信心”当作正确性

Vibe Coding 最危险的时刻不是模型报错，而是模型看起来很自信，但你没有验证。你至少要会看 5 类反馈：终端错误、浏览器报错、网络请求、页面现象、测试结果。

终端层

这里告诉你：命令有没有运行起来、依赖有没有装好、脚本有没有报错、测试有没有通过。

$ npm run dev
$ npm test
$ git diff
Error: Cannot find module ...
浏览器层

这里告诉你：页面是否渲染、按钮是否生效、控制台是否报错、接口是否请求成功。

  • 看 Console：JavaScript 报错。
  • 看 Network：请求是否发出、状态码是否正常。
  • 看 Elements：DOM 是否真的被渲染。
验证层

至少要验证 4 件事：代码 diff、页面结果、测试结果、关键路径。不是“看起来像对”就算结束。

diff→ 运行→ 测试→ 提交
看到错误后先问的 5 个问题
  1. 这是终端错误、浏览器错误、网络错误，还是逻辑错误？
  2. 错误发生在启动时、点击后、提交后，还是渲染时？
  3. 是路径问题、变量问题、类型问题、依赖问题，还是权限/环境变量问题？
  4. 我能不能先缩小范围，复现出最小问题？
  5. 修复后我用什么证据证明它真的好了？
最值得先学会的调试动作
console.log / console.error

先把关键变量和流程点打印出来，看它到底有没有值、值长什么样、代码有没有走到这里。

console.trace()

当你不知道某段逻辑为什么会被执行时，看调用链很有帮助。

浏览器 Network 面板

接口有没有发出去、返回什么状态码、回来的 JSON 长什么样，这里最直观。

git diff

当你怀疑 AI 改坏了东西时，先看 diff，不要凭记忆猜。

测试分层：不要只会看绿色勾

单元测试检查一个函数或一个小模块；集成测试检查几个模块连起来后是否正确；端到端测试检查真实用户路径能不能走通。

AI 改完代码后最少验证什么
  • 关键函数有没有返回正确结果。
  • 主要页面能不能加载和点击。
  • 核心接口有没有正常请求和返回。
  • 改动有没有破坏旧功能。
最小案例 
function add(a, b) {
  return a + b;
}

add(1, 2) === 3;

这就是最小的“验证思维”。先验证最小逻辑，再验证页面和流程，不要一开始就把所有问题混成一团。

Module 8

案例拆解：把概念放进真实 Vibe Coding 场景里

下面这些案例不是“标准答案”，而是帮你建立判断顺序。你要学会的不是某一条命令，而是每种问题该从哪一层开始排查。

案例 1：按钮文案改了，但页面没变

场景：你让 AI 改首页按钮文案，代码看起来改了，但浏览器里还是旧内容。

应该怎么查

先看你改的是不是正确文件；再看开发服务器是否重启；然后看浏览器缓存、热更新是否生效；最后看有没有多个同名组件。

案例 2：列表页空白，接口看似没问题

场景：浏览器没有报红，但页面就是空。AI 说“接口已经返回成功”。

应该怎么查

打开 Network 看响应 JSON 结构；再检查前端代码是否按正确字段取值，例如 data.items 还是 data.results；最后看 map 循环有没有跑到。

案例 3：登录接口 401 / 403

场景：功能逻辑好像没错，但请求总是未授权。

应该怎么查

先看环境变量里的 Key/Token 是否存在；再看请求头有没有带上；再看是不是请求到了错误环境或错误域名。

案例 4：npm run dev 启动失败

场景：AI 让你运行项目，但命令一执行就报错。

应该怎么查

先确认当前目录是否是项目根目录；再看 package.json 是否存在对应脚本；再看依赖是否已经 npm install；最后看 Node 版本是否匹配。

案例 5：AI 一次改了太多，结果更乱

场景：一个简单需求，AI 却动了很多文件，页面还出现回归。

应该怎么查

先用 git diff 看改动边界；把任务拆小；只保留必要改动；必要时回退一部分文件或重新建分支重做。

案例 6：测试通过了，但页面还是坏的

场景：终端里 tests pass，但真实点击流程不对。

应该怎么查

说明测试覆盖面不足。要补页面层验证：浏览器操作、关键路径点击、接口响应、渲染结果，而不是只看单元测试绿了。

Module 9

AI 编程工作流：你不该只会“让模型改代码”

Claude Code 和 Codex 真正值钱的用法，不是让它们盲写，而是你能带着一个稳定流程去驱动它们。流程一旦对，模型换了也能继续工作。

最推荐的闭环
读需求→ 读代码→ 改代码→ 运行→ 看报错→ 修复→ 验证→ commit / PR
  • 读需求：你到底要改什么，不要直接跳实现。
  • 读代码：先看目录、关键文件、函数入口和数据流。
  • 运行与报错：不要只靠“看起来像对”。
  • 验证：至少看 diff、看测试、看页面、看输出。
你现在最容易踩的坑
直接让 AI 改，但自己不看 diff

这样你不知道它到底改了哪些文件、顺手动了什么逻辑、有没有引入无关改动。

看到报错就继续追问模型，不先读错误信息

报错里通常已经告诉你是路径错、变量未定义、模块找不到、类型不对还是权限不足。

没有验证就 commit

只改不跑、不看页面、不看测试，是最容易把“AI 幻觉”带进仓库的方式。

不拆任务

越大的改动越应该拆成小步骤，不然模型很容易一次动太多地方。

Module 11

术语卡：高频词一次讲明白

点击展开下面的术语卡。每张卡都包含白话解释、真实场景，以及它和 AI 编程的关系。

Repository / 仓库

仓库是项目的完整容器，里面不仅有文件，还有分支、历史、标签和说明文档。白话理解：项目的正式档案柜。AI 改的是文件，但协作发生在仓库这个边界里。

Clone / 克隆

把远程仓库完整复制到你本地，包括历史记录。不是下载 zip，而是把整个项目历史一起拿下来。

Branch / 分支

分支是一条独立工作线，让你在不影响默认分支时尝试功能或修 bug。AI 改动大时，分支是最重要的安全隔离层。

Commit / 提交

commit 是一次带说明的版本快照。它不是上传，而是在本地写下一条清楚的历史记录。

Push / 推送

把本地 commit 发送到远程仓库。commit 只在你电脑里，push 后 GitHub 才看得见。

Pull / 拉取

把远程仓库的最新变化获取到本地。你不是一个人在改，所以本地得跟上远程节奏。

Merge / 合并

把一个分支上的变化合并到另一个分支。支线剧情结束，把成果并回主线。

Pull Request / PR

PR 是请求把你的改动合并到目标分支，并邀请别人 review。PR 是协作窗口，不是一个本地命令。

Fork / 派生

把别人的仓库复制到你自己的 GitHub 账号下。参与开源项目但没有写权限时，fork 是常见起点。

Issue / 问题单

Issue 用来记录 Bug、需求、讨论点和待办事项。它不是代码，但它决定你为什么改代码。

README / 项目入口说明

README 是别人理解仓库的第一站。没有 README 的仓库，别人打开后常常不知道该干什么。

GitHub Actions / 自动化

Actions 是 GitHub 上的自动化能力，可在 push、PR、发布等时机自动运行测试、构建、部署或检查。

Module 12

命令工作台：你最先该理解哪些 Git 指令

不需要一口气背完全部命令。先抓住最常用的那几条，理解它们在真实工作流里的位置。

git status

先看状态，再决定下一步。它会告诉你哪些文件被修改、哪些已暂存、当前在哪个分支。让 AI 改完代码后，这通常是第一条该看的命令。

git clone <repo-url>

第一次把远程仓库完整拉到本地。你要在本地跑项目时，通常从 clone 开始。

git switch -c feature/my-task

创建并切换到新分支。开始一个新功能、修一个 bug、或让 AI 尝试大改动时都应该先做这一步。

git add <file>

把改动放入暂存区。AI 改动常常溢出范围，add 能帮你把提交粒度收紧。

git commit -m 'message'

写下一个清楚的历史节点。message 要写人话，说明这次改动解决了什么。

git diff

真正理解改动内容。diff 会精确显示哪些行被删除、哪些行被新增。真正可信的是 diff，不是“AI 说它改好了”。

git pull

把远程最新变化同步到本地。让 AI 开始改之前先 pull，可减少基于旧代码工作的概率。

git push -u origin <branch>

把本地成果发到 GitHub。push 后 GitHub 才能看到你的分支与 commit，PR 也才能开起来。

git log --oneline

快速查看历史。回忆最近变更、定位问题引入时间时很实用。

最应该记住的顺序

git status → git diff → git add → git commit → git push → 开 PR。

Module 13

GitHub Flow：最适合新手的一条协作流程

GitHub 官方长期推荐 GitHub Flow 作为轻量、基于分支的工作方式。先把这条跑顺，比先学复杂流程更划算。

1. 建分支

别直接在默认分支上乱改，先开一条工作分支。

2. 改代码

在分支里完成一组相关修改，范围保持集中。

3. 提交并推送

用清晰 commit message 记录，再同步到远程。

4. 开 PR

请求 review，也可以先开 Draft PR 提前拿反馈。

5. 合并与清理

review 通过后合并，并删除已完成的分支。

纸上练习

请先不要看答案，在心里或纸上写出正确顺序：创建分支、修改代码、git add、git commit、git push、创建 PR、通过 review、合并 PR。

展开查看标准顺序

正确顺序是：创建分支 → 修改代码 → git add → git commit → git push → 创建 PR → 通过 review → 合并 PR。

Module 16

快速测验

先自己回答，再展开答案。这样比直接看解释更容易真正记住。

1. Git 和 GitHub 有什么区别？

Git 是版本控制工具；GitHub 是托管仓库与协作平台。

2. commit 是不是已经上传到 GitHub？

不是。commit 只发生在本地，push 后 GitHub 才能看到。

3. PR 最准确的意思是什么？

请求把你的改动合并到目标分支，并邀请 review。

4. 看到 file not found 时第一反应应该检查什么？

当前目录、路径写法、文件是否真的存在，而不是先猜模型坏了。

5. 一个网页请求大致会经过哪几层？

浏览器/前端 → 后端/API → 数据库 → 返回结果。

6. AI 改完代码后第一条最值得看的命令是什么？

git status，然后通常会接着看 git diff。

7. 为什么 AI 编程更需要 Git？

因为 AI 让改动更快，而 Git 让这些改动可审查、可回退、可协作。

8. 为什么函数值得你现在就搞懂？

因为它是“输入 → 处理 → 输出”的最小抽象单元，很多代码理解都能先从函数开始。
